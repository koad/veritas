## Claim 1: arxiv:2501.09674 — MIT Media Lab / ICML paper on delegation

**Post states:**
> "From arxiv:2501.09674 (*Authenticated Delegation and Authorized AI Agents*, South, Marro, Hardjono et al., ICML 2025)"
> "By the third delegation hop, there is no cryptographic link to the initiating agent or user."

**Findings:**

The paper exists and the authors are correctly named (South, Marro, Hardjono, Mahari, Whitney, Greenwood, Chan, Pentland).

**ICML 2025: UNVERIFIED / LIKELY INACCURATE.** The paper was submitted to arXiv on January 16, 2025. The HTML version of the paper carries a "Machine Learning, ICML" classification tag, but no confirmation of acceptance as a published ICML proceeding was found. It appears to be an arXiv preprint. Calling it "ICML 2025" overstates its publication status unless acceptance is confirmed.

**MIT Media Lab: PARTIALLY ACCURATE.** The paper does not list institutional affiliations on the arXiv abstract page. However, Thomas Hardjono is the Technical Director of MIT's Internet Trust Consortium, and Alex Pentland is a professor at MIT Media Lab — these are the most recognizable names on the paper. Attributing the paper to "MIT Media Lab" is an imprecise shorthand; the correct attribution would be MIT Connection Science / MIT Internet Trust Consortium. The "MIT" connection is real; "MIT Media Lab" as the named institution is an approximation.

**"Third delegation hop" quote: NOT FOUND IN THE PAPER.** A full-text search of the HTML version of arxiv:2501.09674 found no instance of the phrase "third delegation hop" or "no cryptographic link to the initiating agent or user." The paper does address multi-hop delegation (Section 4.4.1, "Inter-agent scoping") and discusses maintaining cryptographic accountability in delegation chains, but the specific quoted language does not appear in the paper. This quote should not be presented as a direct citation from this source.

**Action required:** Remove the quotation marks and citation attribution for that specific phrase, or locate the actual source. The general concept (delegation chains losing cryptographic provenance) is sound and is addressed in this literature, but the specific quoted text cannot be verified as originating from arxiv:2501.09674.

## Claim 2: CSA 97% excess-privilege figure

**Post states:**
> "The Cloud Security Alliance quantified the failure mode in March 2026: 97% of non-human identities carry excess privileges."

**Findings:**

The 97% figure is real, but the attribution is wrong. The statistic originates from **Entro Security's "2025 State of Non-Human Identities and Secrets in Cybersecurity" report**, released September 2024 — not from CSA in March 2026. The CSA's March 2026 report ("The State of Cloud and AI Security in 2026") does not contain a 97% figure; it reports a 100-to-1 ratio of machine-to-human identities, sourced from Tenable's research.

The CSA's own January 2026 Non-Human Identities report (with Oasis Security) reports that 33% of organizations flagged overprivileged NHIs as a top risk, and 37% of NHI incidents stemmed from overprovisioned access — different figures entirely.

**Action required:** Correct the source attribution. The 97% figure is from Entro Security (September 2024), not from CSA (March 2026). Faber can still use the statistic, but must cite it accurately.

## Claim 4: Biscuit token claims

**Post states:**
> "Biscuits are a public-key version of macaroons — Ed25519-chained signatures with Datalog policies that express conditional delegation. The holder can add constraints that reduce authority but cannot expand it. Verification does not require the root secret."

**Findings:**

**ACCURATE.** All three technical claims are correct:
- Biscuit uses Ed25519 (also ECDSA over P256) signatures. ✓
- Uses a modified Datalog for policy evaluation. ✓
- Supports offline attenuation — holder can add restrictions, cannot expand authority. ✓
- Verification requires only the public key, not the root secret. ✓

The distinction from macaroons (public key vs. shared secret MAC) is correctly stated.

The arxiv:2603.24775 citation (AIP paper) is real and verified — it does build on Biscuit tokens for multi-hop delegation in MCP/A2A protocols, introducing Invocation-Bound Capability Tokens (IBCTs).

**Post states:**
> "Their NCCoE concept paper adapts SP 800-53 controls for AI agents, specifically naming: Multi-agent trust boundaries, Chain-of-custody logging for autonomous operations, Least-privilege tool access, Agent action containment"

**PARTIALLY ACCURATE.** NIST's AI Agent Standards Initiative was announced February 17, 2026, and the URL nist.gov/caisi is correct. The COSAiS project (SP 800-53 Control Overlays for Securing AI Systems) at csrc.nist.gov/projects/cosais does explicitly adapt SP 800-53 for AI agent systems, including single-agent and multi-agent use cases. The announcement date (February 17, 2026) is correct.

However, the four specific bullet points (multi-agent trust boundaries, chain-of-custody logging, least-privilege tool access, agent action containment) cannot be verified as appearing verbatim in the NCCoE concept paper. The COSAiS project addresses these themes, but the exact four-item list may be a synthesis rather than a direct quote. The framing should be "these themes are addressed" rather than "specifically naming" unless Faber has the concept paper and can confirm exact language.

**Action required:** Minor — soften "specifically naming" to "addressing" unless exact sourcing from the concept paper is available.

## Claim 6: 81% attack success rate

**Post states:**
> "NIST red-team exercises (January 2025) reported an 81% attack success rate against AI agents versus 11% against baseline defenses. The vector was trust boundary weaknesses — agents accepting instructions from other agents without verifying authorization."

**Findings:**

The 81% and 11% figures are **ACCURATE** — they appear in NIST's January 2025 technical blog "Strengthening AI Agent Hijacking Evaluations," which reported an increase in attack success rate from 11% (strongest baseline attack) to 81% (strongest new red-team attack).

However, the characterization of the attack vector is **INACCURATE**. The actual vector described by NIST is **agent hijacking / indirect prompt injection** — where an agent processes malicious data (emails, files, websites) containing instructions that hijack the agent's behavior. The three categories tested were: remote code execution via agent tool use, database exfiltration, and automated phishing.

The post's framing — "agents accepting instructions from other agents without verifying authorization" — describes a trust boundary / inter-agent authorization failure, which is a different, related but distinct problem. NIST's red-team exercises specifically targeted prompt injection / data-instruction conflation, not agent-to-agent authorization verification failures.

**Action required:** Correct the attack vector description. The 81%/11% numbers are real; the attack type described is not what those numbers measure.

## Claim 8: GPG properties — "non-repudiation, decentralized, offline-capable, 30-year production history"

**Post states:**
> "GPG clearsign provides non-repudiation... It is decentralized — there is no certificate authority... It is offline-capable... battle-tested across thirty years of production use"

**Findings:**

**ACCURATE.** All four properties are correctly characterized:
- **Non-repudiation:** Correct — the signer cannot plausibly deny having signed with their private key.
- **Decentralized:** Correct — GPG uses a Web of Trust model with no mandatory central CA.
- **Offline-capable:** Correct — verification requires only the public key, which can be distributed statically.
- **30-year production history:** Correct — GnuPG was released in 1999, and the OpenPGP standard (RFC 2440) dates to 1998. As of 2026, that is approximately 27-28 years, rounding to "thirty years" is a reasonable characterization.
