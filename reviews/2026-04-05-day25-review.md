## 1. Lampson 1983 — Citation Verification

**Verdict: Accurate, with a framing nuance worth noting.**

Butler Lampson's "Hints for Computer System Design" is a real and widely cited paper. Publication details:

- **Full citation:** Lampson, B. W. (1983). "Hints for Computer System Design." *ACM SIGOPS Operating Systems Review*, 17(5), pp. 33–48. Also appeared in *Proceedings of the 9th ACM Symposium on Operating Systems Principles (SOSP '83)*.
- **ACM DL:** https://dl.acm.org/doi/10.1145/773379.806614
- **Turing Award:** Lampson received the ACM Turing Award in 1992. The paper is routinely cited in the award context but was published nine years before the award, not as the Turing submission.

**The specific principle:** Faber's phrasing — "match the storage substrate to the dominant access pattern, not to the most general-purpose tool available" — is a reasonable paraphrase of Lampson's broader hints about storage and abstraction selection. Sibyl's research notes that the direct quote is "Use a different abstraction for storing data than for processing it," with the access-pattern-matching principle as the practical implication. Faber's paraphrase is accurate in spirit and consistent with Lampson's documented advice.

**Minor note:** The post says "the paper the Turing committee cited when they gave him the award." The Turing Award was given in 1992 for a broad body of work, not specifically for this 1983 paper. The post's framing implies the 1983 paper was the specific basis for the award. This is a small overstatement — the paper is frequently associated with the award but was not the sole or explicit citation. Recommend softening to: "Butler Lampson's 1983 paper — one of the foundational texts in systems design" rather than implying the award was specifically for this paper.

## 2. Gray & Reuter 1992 — Citation Verification

**Verdict: Accurate.**

Jim Gray and Andreas Reuter's *Transaction Processing: Concepts and Techniques* (1992) is a real book and remains the reference text on database transaction theory.

- **Full citation:** Gray, J. & Reuter, A. (1992). *Transaction Processing: Concepts and Techniques.* Morgan Kaufmann Publishers. ISBN 1-55860-190-2.
- Gray received the ACM Turing Award in 1998, substantially for this work and related contributions to transaction processing.

**The specific claim:** Faber attributes to Gray & Reuter the principle that concurrent multi-writer access to shared mutable state requiring ACID guarantees is the condition that justifies a full database engine. This is accurate and consistent with the book's documented content — ACID properties (Atomicity, Consistency, Isolation, Durability) and the conditions under which they are necessary are the central subject matter.

**No issues.** The characterization is accurate and the claim is defensible against the source.

## 3. Kleppmann 2017 — Citation Verification

**Verdict: Accurate.**

Martin Kleppmann's *Designing Data-Intensive Applications* (2017) is a real book published by O'Reilly Media.

- **Full citation:** Kleppmann, M. (2017). *Designing Data-Intensive Applications: The Big Ideas Behind Reliable, Scalable, and Maintainable Systems.* O'Reilly Media. ISBN 978-1-4493-7332-0.

**Chapter 3 claim:** Faber states that Chapter 3 ("Storage and Retrieval") argues that log-structured storage outperforms B-tree page-oriented storage for write-heavy workloads where update-in-place is not required. This accurately reflects Chapter 3's documented content, which covers SSTables, LSM-trees (Log-Structured Merge-Trees), and compares them to B-tree storage. The access-pattern-dependent performance tradeoff is a central claim in that chapter.

**Chapter 10 claim:** Faber states Chapter 10 describes the event-sourcing pattern — current state as a view over an immutable log. Chapter 10 of DDIA covers "Batch Processing" and does discuss immutable data and event logs, though the fuller event-sourcing treatment appears across Chapters 10–11 (stream processing). The attribution is directionally correct; the precise chapter reference is slightly loose but not materially wrong.

**No issues requiring correction.**

## 5. MongoDB Passengers Collection — Technical Description

**Verdict: Partially accurate. One factual correction required.**

The post describes the Passengers collection as handling "real-time presence data, active session presence, live connection status" with "multiple writers (multiple entity processes, daemon workers, external event listeners)."

**What the source code actually shows:**

The Passengers collection is defined in `/home/koad/.koad-io/daemon/src/server/passenger-methods.js` as:

```javascript
const Passengers = new Mongo.Collection('Passengers', {connection: null});
```

The `{connection: null}` parameter is significant: it means this collection is **server-side only** and is **not published via DDP to clients**. The Passengers collection is populated by scanning entity directories for `passenger.json` files and reading entity `.env` files — it is a server-side registry of registered entity identities, not a real-time presence or connection-tracking collection.

The real-time session tracking in the daemon is in the `ApplicationSessions` collection (via `/home/koad/.koad-io/packages/koad-io/server/connection-tracker.js`), and the live connection state is tracked there, not in Passengers.

**What the post gets right:** MongoDB does handle real-time shared operational state in the daemon. The workers collection (`WorkerProcesses = new Mongo.Collection('workers')`) and the `ApplicationEvents`, `ApplicationProcesses` collections in the core package handle live daemon state. The git vs. MongoDB split is correctly characterized at the architectural level.

**What requires correction:** The Passengers collection is more accurately described as a server-side entity registry (populated from `passenger.json` files on disk) than a real-time presence or live connection collection. The post's framing — "live entity presence, active session presence, live connection status" as Passengers — is imprecise. The live connection data lives in ApplicationSessions.

**Recommended correction (minor):** In the architecture diagram, label `passengers collection` as "entity registry / display configuration" rather than "live entity presence." Or generalize the MongoDB section to say "sessions collection, workers collection, events collection" and note that Passengers provides entity configuration. The core argument — MongoDB for multi-writer shared state, git for identity and history — remains correct.

## 6. Git as Content-Addressable Storage

**Verdict: Accurate.**

The post claims git provides:
1. Content-addressable storage (SHA hash = content identity)
2. Append-only history (objects not overwritten absent `git gc`)
3. Atomic commits
4. Distributed replication via push/pull
5. Branching

All five claims are accurate and consistent with git's documented design. The content-addressable property (every object identified by SHA hash of its content) is fundamental to git's object model and is correctly characterized. The note about SHA-1 transitioning to SHA-256 is documented in git's hash-function-transition documentation.

The claim that "you cannot modify an object without changing its address" is accurate by design — this is the core property of content-addressable storage.

The post correctly notes this is "single-writer atomicity, not full ACID isolation" — an accurate and important qualification.

**No issues.**

## 7. Other Factual Claims

**MongoDB + DDP architecture:** The post correctly identifies that the daemon uses DDP (Distributed Data Protocol) and that MongoDB is the backing store. DDP is confirmed in multiple files under `/home/koad/.koad-io/passenger/dist/background/ddp-connection.js` and related. The daemon is Meteor-based, which mandates MongoDB as the backing store — the post doesn't mention this implementation detail, but the description is accurate.

**Workers collection:** The post lists a `workers collection` in its architecture diagram. This is confirmed: `WorkerProcesses = new Mongo.Collection('workers')` in `/home/koad/.koad-io/packages/workers/server/collections.js`.

**Events collection:** The post lists an `events collection`. Confirmed: `ApplicationEvents = new Mongo.Collection('events', koad.mongo)` in `/home/koad/.koad-io/packages/core/server/collections.js`.

**"No sub-commit transactions":** Accurate. The post correctly identifies that filesystem writes below the git commit level lack transactional semantics and recommends the `mv`-as-atomic-write mitigation. This is standard POSIX behavior and the mitigation is correct.

**"Operational data store vs. record of system":** This is a real and well-established pattern in data architecture. The characterization is accurate.
