## 6. GnuPG Deprecation of Clearsign

**Claim:** "GnuPG's own documentation also deprecates clearsign in favor of detached signatures — more robust against edge cases in document encoding."

**Verdict: Accurate, and stronger than the post implies.**

This is confirmed by GnuPG's official man page (`gpg.1`) and operational documentation, which state: "It is suggested to avoid cleartext signatures in favor of detached signatures." The reason given is that cleartext signature verification only covers the signed data — other content in the file is not verified, creating edge-case pitfalls. This is precisely what the post says.

The post's framing ("also deprecates") softens what is a direct recommendation in GnuPG's own documentation. "Deprecates" is the right word and the documentation supports a stronger characterization. The post is accurate; it errs toward understatement.

## 7. GPG Technical Claims

**Claim:** "Modern GPG (2.1+) auto-generates a revocation certificate at key creation, stored in `~/.gnupg/openpgp-revocs.d/`."

**Verdict: Accurate.** Confirmed by Sibyl's research (GnuPG documentation, Arch Linux wiki sources) and consistent with GnuPG 2.1+ behavior.

**Claim:** "GPG revocation is: publish a signed certificate and the key is globally invalid, verifiable by anyone, without contacting the original issuer."

**Verdict: Accurate** as a description of the model. One nuance: revocation is only verifiable by parties who retrieve the updated key from a keyserver or are otherwise informed. "Globally invalid" is accurate in that any verifier who checks the keyserver will see the revocation, but unlike a centralized system, propagation is not instantaneous or guaranteed to reach every party. The post's wording is standard for describing GPG revocation and does not materially overstate.

## 8. koad:io Technical Claims

### Bond structure field enumeration

**Claim:** "A bond between koad and Juno is a structured markdown file with six fields: grantor, grantee, scope, domain, issued date, and signature."

**Verdict: Inaccurate. Requires correction.**

The actual `koad-to-juno.md` front matter contains: `type`, `from`, `to`, `status`, `visibility`, `created`, `renewal`. The actual bond structure does not use the field names "grantor," "grantee," "scope," or "domain" as named front matter fields. The scope and domain are described in the body prose, not enumerated as discrete labeled fields.

A more accurate description of what the bond document contains:
- Front matter: `type` (bond type: authorized-agent, peer, etc.), `from` (grantor identity), `to` (grantee identity), `status`, `visibility`, `created`, `renewal`
- Body: bond statement, authorized actions (scope), trust chain, signing block, revocation terms

The post's "six fields" enumeration lists conceptual elements correctly but names them with labels that do not match the actual field names in the documents. This matters because the post is describing the koad:io implementation specifically. A developer reading this and then examining the actual bond file will find a mismatch.

**Recommended correction:** Replace "a structured markdown file with six fields: grantor, grantee, scope, domain, issued date, and signature" with something like: "a structured markdown file with front matter recording bond type, parties (from/to), creation date, and status — with the authorized scope spelled out in the body."

### Public key location

**Verdict: Accurate.** This is consistent with CLAUDE.md documentation ("`juno.keys` → `canon.koad.sh/juno.keys`") and standard koad:io entity key distribution pattern.

### Ed25519 key usage

**Verdict: Accurate.** Confirmed by the presence of `/home/koad/.juno/id/ed25519` and `ed25519.pub`. Ed25519 is the primary key type for entity identity.

## 10. JWT/OAuth Technical Claims

**Claim:** JWT asserts that "the issuer believed these claims at the time of signing. It does not assert the claims are still true. And it cannot be revoked without external infrastructure."

**Verdict: Accurate.** This is a correct description of JWT semantics per RFC 7519. Sibyl rates this high confidence as "the RFC 7519 specification."

**Claim:** The three revocation workarounds enumerated (blacklist, short expiry + refresh, token versioning) each reintroduce server state or do not constitute true revocation.

**Verdict: Accurate.** This is a well-documented limitation of stateless JWTs. The characterization that none of these are "revocation in the meaningful sense" is analytically correct and is a standard position in security literature.
